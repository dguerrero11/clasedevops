01

¿Qué es GitOps?

Git como única fuente de verdad para infraestructura y aplicaciones

Los 4 principios de GitOps

📁 Declarativo

El estado deseado del sistema se describe en archivos YAML. No scripts. No comandos. Archivos versionados.

🔢 Versionado

Git es el único lugar donde vive la verdad. Cada cambio tiene historial, autor, fecha. Rollback = revert commit.

🤖 Automatizado

Los cambios aprobados en Git se aplican solos al cluster. No hace falta hacer kubectl apply manualmente.

🔄 Reconciliado

Un agente (ArgoCD) vigila constantemente que cluster = Git. Si alguien cambia algo a mano, ArgoCD lo revierte.

Regla de oro: Si no está en Git, no existe en producción.
GitOps vs CI/CD tradicional
CI/CD Tradicional (Push)GitOps (Pull)
¿Quién aplica cambios?Pipeline hace kubectl applyArgoCD detecta y aplica solo
Credenciales del clusterEn el pipeline (riesgo de exposición)Solo ArgoCD las tiene
RollbackRe-ejecutar pipeline con tag anteriorRevert en Git o clic en UI
Drift detectionNo existeArgoCD alerta en segundos

⚙️ CI — Tekton (build)

  • Task git-clone: clona el repo en workspace NFS
  • Task build-push: Kaniko construye la imagen
  • Pushea a Docker Hub como :v1 o :v2

🔄 CD — ArgoCD (deploy)

  • Monitorea k8s/ en GitHub cada 3 min
  • Detecta el nuevo tag en deployment.yaml
  • Aplica los manifiestos al namespace demo
La separación app/ vs k8s/

📝 app/ — El Dev lo toca

  • index.html — código fuente
  • Dockerfile — cómo construir la imagen

Tekton lee esto para construir la imagen.

🗂️ k8s/ — El Ops lo toca

  • namespace.yaml
  • deployment.yaml ← el tag de imagen
  • service.yaml

ArgoCD solo mira esta carpeta.

⚠️ Regla crítica: Si el Dev cambia app/index.html pero NO actualiza el tag en k8s/deployment.yaml, ArgoCD no despliega nada. El tag es la señal.

Tekton Pipelines

CI nativo de Kubernetes — construye imágenes dentro del cluster

Componentes de Tekton
  • T
    Task
    Unidad mínima. Define Steps (contenedores). Tiene Inputs, Outputs y Workspaces.
  • P
    Pipeline
    Encadena Tasks en orden. Comparte Workspaces entre ellas. Define params globales.
  • PR
    PipelineRun
    Instancia de ejecución. Aporta los valores reales de params y crea el PVC para el workspace.
# Una Task: dos Steps
apiVersion: tekton.dev/v1beta1
kind: Task
spec:
  workspaces:
    - name: source
  steps:
    - name: git-clone
      image: alpine/git
    - name: build
      image: gcr.io/kaniko-project/executor
Cada Step es un contenedor independiente. El Workspace (PVC NFS) los conecta.
¿Por qué Kaniko?

❌ El problema

Construir imágenes Docker dentro de Kubernetes requiere acceso al Docker socket (/var/run/docker.sock). Eso exige privilegios de root en el nodo — inaceptable en producción.

✅ La solución: Kaniko

Construye imágenes desde un Dockerfile en userspace, sin Docker daemon. No necesita acceso al socket. Funciona como un contenedor normal en Kubernetes.

# Task build-push usa Kaniko
steps:
  - name: build-and-push
    image: gcr.io/kaniko-project/executor:latest
    args:
      - --dockerfile=$(workspaces.source.path)/app/Dockerfile
      - --context=$(workspaces.source.path)/app
      - --destination=$(params.image)
    volumeMounts:
      - name: docker-credentials
        mountPath: /kaniko/.docker
⚠️ PodSecurity: El namespace tekton-pipelines debe tener la label enforce=privileged o Kaniko no arranca.
El Workspace compartido — NFS PVC
⚙️ Task: clone
Pod 1
escribe código → /workspace/source
📦 PVC NFS
500Mi nfs-csi
🔨 Task: build-push
Pod 2
lee código desde → /workspace/source
📦 PVC NFS
mismo PVC
¿Por qué NFS y no emptyDir? Cada Task corre en un Pod diferente en nodos diferentes. emptyDir vive solo en un nodo. NFS es accesible desde cualquier nodo del cluster — por eso el PVC usa storageClassName: nfs-csi.
🛠️ Demo en vivo
Instalar Tekton
# 1. Instalar Tekton Pipelines
kubectl apply -f https://storage.googleapis.com/tekton-releases/pipeline/latest/release.yaml
kubectl get pods -n tekton-pipelines --watch

# 2. ⚠️ CRÍTICO: etiquetar namespace (PodSecurity)
kubectl label namespace tekton-pipelines \
  pod-security.kubernetes.io/enforce=privileged --overwrite

# 3. Instalar CLI tkn
curl -LO https://github.com/tektoncd/cli/releases/download/v0.35.0/tkn_0.35.0_Linux_x86_64.tar.gz
tar xzf tkn_0.35.0_Linux_x86_64.tar.gz -C /usr/local/bin tkn
tkn version

# 4. Secret Docker Hub
kubectl create secret docker-registry docker-credentials \
  --docker-server=https://index.docker.io/v1/ \
  --docker-username=TU_USUARIO \
  --docker-password=TU_TOKEN \
  -n tekton-pipelines

04

ArgoCD

CD declarativo — el cluster siempre igual a Git

ArgoCD — Conceptos clave
  • A
    Application
    Objeto de ArgoCD que vincula un repo Git con un namespace del cluster. Define qué monitorear y dónde desplegarlo.
  • S
    selfHeal: true
    Si alguien hace kubectl delete deployment manualmente, ArgoCD lo restaura en ~10 segundos.
  • P
    prune: true
    Si un archivo se borra de Git, ArgoCD borra el recurso del cluster. Git = fuente de verdad absoluta.
spec:
  source:
    repoURL: https://github.com/...
    path: k8s          # ← solo mira esta carpeta
    targetRevision: main
  destination:
    namespace: demo
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
prune: true — si borras un YAML de Git, ArgoCD destruye el recurso del cluster. Úsalo con cuidado en producción.
🛠️ Demo en vivo
Instalar ArgoCD
# 1. Instalar
kubectl create namespace argocd
kubectl apply -n argocd -f \
  https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
kubectl get pods -n argocd --watch

# 2. Exponer con NodePort
kubectl patch svc argocd-server -n argocd \
  -p '{"spec":{"type":"NodePort","ports":[{"port":443,"nodePort":30443,"targetPort":8080,"protocol":"TCP","name":"https"}]}}'

# 3. Obtener contraseña
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d && echo

# 4. Abrir en el navegador
# https://192.168.109.143:30443
# Usuario: admin  |  Password: (el del paso anterior)

selfHeal + Rollback

El cluster siempre vuelve a lo que dice Git

💥 Demo en vivo — selfHeal
¿Qué pasa si borro algo a mano?
# Escenario 1: borrar un pod
kubectl delete pod -l app=page-demo -n demo
kubectl get pods -n demo -w
# ArgoCD lo restaura en ~10 segundos

# Escenario 2: borrar el Deployment completo
kubectl delete deployment page-demo -n demo
kubectl get deployment -n demo -w
# ArgoCD lo restaura en ~10 segundos

# Escenario 3: cambiar replicas a mano
kubectl scale deployment page-demo -n demo --replicas=5
kubectl get pods -n demo -w
# ArgoCD lo revierte a 2 réplicas (lo que dice Git)
selfHeal: true — ArgoCD es el guardián. Lo que no está en Git, no sobrevive.
Rollback — 3 formas

🖱️ ArgoCD UI

GitOps puro

  1. page-demo
  2. HISTORY AND ROLLBACK
  3. Seleccionar commit v1
  4. Rollback → OK

📝 Git revert

Recomendado

git revert HEAD
git push
# ArgoCD sync automático

⚡ kubectl rollout

No GitOps

kubectl rollout undo \
  deployment/page-demo \
  -n demo
# ArgoCD revertirá de vuelta

selfHeal lo deshace en ~10s
