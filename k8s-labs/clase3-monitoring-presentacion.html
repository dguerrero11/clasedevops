02

Loki + Promtail

Logs de todos los pods del cluster — demo, tekton, argocd — en Grafana

Desplegar Loki + Promtail
# Prerequisito: etiquetar namespace
kubectl label namespace monitoring \
  pod-security.kubernetes.io/enforce=privileged \
  --overwrite

# 1. Loki (servidor de logs)
kubectl apply -f loki/loki-configmap.yaml
kubectl apply -f loki/loki-pvc.yaml
kubectl apply -f loki/loki-deployment.yaml
kubectl apply -f loki/loki-service.yaml

# 2. Promtail (colector en cada nodo)
kubectl apply -f promtail/promtail-rbac.yaml
kubectl apply -f promtail/promtail-configmap.yaml
kubectl apply -f promtail/promtail-daemonset.yaml

# Verificar
kubectl get pods -n monitoring -l app=loki
kubectl get pods -n monitoring -l app=promtail

🔄 Cómo funciona

  • Promtail corre en cada nodo como DaemonSet
  • Lee /var/log/pods/* — todos los contenedores
  • Agrega labels: namespace, pod, container
  • Envía a Loki via push HTTP
  • Loki guarda en PVC NFS (5Gi, 7 días)
Si hay error de permisos NFS:
ssh root@192.168.109.210 "chown -R 10001:10001 /srv/nfs/k8s"
kubectl rollout restart deployment/loki -n monitoring
📄 Demo 3 — Loki
Buscar logs de la app y de ArgoCD
Grafana → Explore → Datasource: Loki
Logs de la app (namespace demo) {namespace="demo"}
Logs del sync de ArgoCD {namespace="argocd"}
Ver logs de un Task de Tekton {namespace="tekton-pipelines"}
Buscar errores en cualquier namespace {namespace=~"demo|argocd"} |= "error"
Logs de un pod específico {pod="page-demo-f74bc7f8d-m9b7q"}
LogQL es como PromQL pero para logs.

03

Tempo — Trazas distribuidas

Ver el camino completo de una petición y dónde tardó

Desplegar Tempo + activar trazas en Tekton
# 1. Desplegar Tempo
kubectl apply -f tempo/tempo-configmap.yaml
kubectl apply -f tempo/tempo-pvc.yaml
kubectl apply -f tempo/tempo-deployment.yaml
kubectl apply -f tempo/tempo-service.yaml

# Verificar
kubectl get pods -n monitoring -l app=tempo

# 2. Activar trazas en Tekton (OTLP → Tempo)
kubectl patch configmap config-tracing \
  -n tekton-pipelines --type merge \
  -p '{
    "data": {
      "backend": "otlp",
      "endpoint": "tempo.monitoring.svc:4317"
    }
  }'

kubectl rollout restart deployment/tekton-pipelines-controller \
  -n tekton-pipelines

# El siguiente PipelineRun enviará trazas a Tempo

🔗 Protocolos que acepta Tempo

  • OTLP gRPC :4317 — Tekton, apps modernas
  • OTLP HTTP :4318 — REST
  • Jaeger HTTP :14268 — Java, Go
  • Zipkin :9411 — Spring Boot

📤 Traza de prueba manual

Si la app no tiene instrumentación, enviar una traza desde curl para demostrar que Tempo funciona.

Ver comando en el README-loki-tempo.md

Conectar Loki y Tempo a Grafana automáticamente
# 1. Aplicar el ConfigMap de datasources
kubectl apply -f grafana-datasources-provisioning.yaml

# 2. Parchear Grafana para montar el ConfigMap
kubectl patch deployment grafana \
  -n monitoring --type=json -p='[
  {
    "op": "add",
    "path": "/spec/template/spec/volumes/-",
    "value": {
      "name": "datasources",
      "configMap": {"name": "grafana-datasources"}
    }
  },
  {
    "op": "add",
    "path": "/spec/template/spec/containers/0/volumeMounts/-",
    "value": {
      "name": "datasources",
      "mountPath": "/etc/grafana/provisioning/datasources"
    }
  }
]'

# 3. Reiniciar Grafana
kubectl rollout restart deployment/grafana -n monitoring
kubectl rollout status deployment/grafana -n monitoring

✅ Verificar en Grafana UI

  • Connections → Data sources
  • Deben aparecer automáticamente:
    Prometheus, Loki, Tempo
  • Sin tocar nada en la UI

🔗 Correlaciones activas

  • Prometheus → Tempo: clic en ejemplar → traza
  • Loki → Tempo: clic en traceID en log → traza
  • Tempo → Loki: desde traza → logs del pod
  • Tempo → Prometheus: desde traza → métricas del servicio
Provisioning automático: Grafana carga el directorio /etc/grafana/provisioning/datasources/ en cada arranque.
